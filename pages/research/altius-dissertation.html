Abstract

ALTIUS is an agency-first AI task engine prototype designed to reduce inertia and convert intent into follow-through. Unlike “answer machines,” ALTIUS is framed as a structured co-agent that prioritizes user ownership, next-step clarity, and ethical constraints (nonpartisan, non-therapeutic, data-minimal).

This write-up formalizes ALTIUS as a parallel dissertation thread: (1) a civic learning tool, and (2) a human–AI interaction study that can generate anonymized process measures of task initiation and completion.

Status: prototype exists as a public page; model behavior is not assumed to run locally. “AI” can be implemented via external APIs or a future local model, but research framing and guardrails apply either way.

1. The Problem: Civic Concern Without Civic Follow-Through

Many students experience civic and academic life as a constant stream of high-level obligations: “I should vote,” “I should show up,” “I should write the paper,” “I should email the committee.” The failure mode is often not ignorance. It is friction: overload, ambiguity, social risk, and the psychology of avoidance.

The practical result is a predictable cycle: the student understands what matters, but stalls at the first step—then shame compounds the stall. This is relevant not only to grades but to democracy: sustained civic engagement depends on repeatable habits of initiation, coordination, and follow-through.

Design target

Make the “start” moment smaller, clearer, and less socially/psychologically costly.

Non-goal

Replace the student’s thinking or decision-making. ALTIUS is not a shortcut for agency; it is a scaffold.

2. Theory of Change: Agency as a Repeatable Loop

ALTIUS is built around an operational definition of agency: the capacity to reliably translate intention into action under real constraints (time, uncertainty, anxiety, competing demands).

The intervention is not “motivation.” It is structure: a short interaction loop that forces clarity, decomposes work, anchors a minimal next step, and then iterates through completion.

The ALTIUS loop (core protocol)

  1. Clarify: Name the avoided task and the real deliverable.
  2. Decompose: Split into atomic steps; separate decision work from mechanical work.
  3. Anchor: Identify the smallest non-zero action that can be done now.
  4. Execute & Reflect: Do it, report completion, adjust plan, repeat.

The key claim is that most “I can’t” moments are actually “I can’t see the next step without pain.”

3. System Design: Constraints, Components, and Interfaces

3.1 Design constraints (hard rules)

  • Agency-first: the user chooses goals; the system scaffolds execution.
  • Nonpartisan: no recommendation toward any party or candidate outcomes.
  • Non-therapeutic: not a mental-health tool; refer out if crisis emerges.
  • Data minimization: if logs ever exist, default to anonymized aggregates.
  • Transparency: users must know what ALTIUS is and is not.

3.2 Components (conceptual architecture)

Front-end interface (current)

A public-facing prototype page that communicates purpose, tone, and the interaction loop. This is the “social contract” layer: what users should expect, and how to use it.

Link: altius.html

Task engine logic (research object)

A prompt protocol that reliably produces: next-step plans, time-boxed actions, and structured reflection. This can be implemented with an external LLM API or a future local model.

The model is the engine; the protocol is the steering wheel.

3.3 Implementation options (non-committal, research-safe)

  • API-based LLM (recommended early): fastest to integrate and test the protocol with real users. Research control comes from prompts + guardrails + measurement design.
  • Local model (future track): more autonomy and customization, but significantly more engineering, compute, and evaluation overhead. Useful only after protocol and evaluation are stable.

Important note on “training our own”

Training a new LLM from scratch is usually unrealistic for a student project. If you ever go local, the sane path is: use an existing open model + carefully scoped fine-tuning (or retrieval) + strict evaluation. But the dissertation value doesn’t depend on owning the model — it depends on the theory, protocol, and measurable outcomes.

5. Methods: Evaluation & Study Plan (Parallel Dissertation Track)

ALTIUS can be evaluated as a minimal-risk human–AI interaction intervention focused on task follow-through. The cleanest design is a pilot on campus with a short time window (e.g., 2–3 weeks), then a scale plan.

5.1 Research questions

  • RQ1: Does ALTIUS increase task initiation rates for high-friction civic/academic tasks?
  • RQ2: Does it increase task completion (or meaningful partial completion) within a set time window?
  • RQ3: Which protocol features correlate most strongly with follow-through (anchoring, time-boxing, reflection)?
  • RQ4: Does it help “cross-pressured” or politically misaligned students engage without social risk spirals?

5.2 Study designs (pick one, or sequence them)

Design 1 — Single-group pre/post pilot (fastest)

  • Recruit volunteers.
  • Baseline: self-reported avoidance + current habits.
  • Intervention: ALTIUS protocol for 1–2 tasks.
  • Post: initiation/completion + user-perceived agency.

Pros: speed. Cons: weaker causal inference.

Design 2 — Randomized encouragement (stronger)

  • Recruit participants into a broader study.
  • Randomly encourage ALTIUS use (with standardized onboarding).
  • Compare outcomes between encouraged vs not-encouraged.

Pros: more causal leverage without heavy manipulation.

Integration with the main dissertation study

ALTIUS can be a substudy nested inside the campus framing/norms experiment: after the survey, students who opt in can use ALTIUS to complete a civic micro-task (e.g., drafting one email, signing up for one event, planning one meeting). This links framing effects to real follow-through behavior.

6. Measures & Outcomes

Measures should be boring, explicit, and hard to fake. The point is not “did they like the vibe,” but “did they move.”

6.1 Primary behavioral outcomes

  • Initiation: did the participant start within 24 hours? (yes/no)
  • Completion: completed within study window? (yes/no)
  • Time-to-start: hours between intention and first action
  • Next-step adherence: did they follow the anchored step (yes/no)

6.2 Secondary outcomes

  • Self-efficacy: “I can influence outcomes here” (campus-specific)
  • Agency perception: “I’m steering; the tool is supporting”
  • Over-reliance risk: indicators of deferring decisions to the system

6.3 Minimal data principle

If you collect any logs, store only what you need (task category + timestamp + completion state), and avoid personal narrative content. If you don’t need logs, don’t create them.

7. Ethics & Guardrails

ALTIUS is designed to be low-risk by design. Still, the ethics need to be explicit, because AI tools can accidentally become manipulative, partisan, or dependency-forming if left unchecked.

Guardrails (non-negotiable)

  • Agency-first language: “You choose; I help you execute.”
  • No partisan outcomes: no candidate/party advocacy.
  • No therapy role: if crisis signals emerge, refer to campus/pro services.
  • Consent clarity: what data is collected (if any), and why.
  • Opt-out always: participation optional, no penalty.

Any campus deployment should be reviewed under IRB rules (or equivalent) before data collection.

8. Limits, Threats, and What This Does Not Prove

  • External validity: campus context may not generalize without replication.
  • Selection: volunteers may already be more motivated than average.
  • Novelty effects: early results may be inflated by “new tool” excitement.
  • Measurement: self-reports are fragile; prioritize behavioral outcomes.
  • Model dependence: effects may vary by model capability; protocol should remain stable.

Appendix: Prompt Protocols (Copy/Paste Templates)

These are designed to be model-agnostic. You can run them manually (in a chat) or embed them later. The important thing is consistency: same structure across participants.

A1. Intake prompt (Clarify)

ALTIUS INTAKE — RULES
You are a task engine. You do not make decisions for me.
You do not write full assignments that should be mine.
You produce: clarity, a plan, and the smallest next action.

TASK:
- What task am I avoiding?
- What's the real deliverable (file/email/meeting outcome)?
- What's the deadline (or "none")?
- Why is this hard (1–2 sentences)?
- What resources do I already have (notes, links, docs)?

A2. Decomposition prompt (Decompose + Anchor)

DECOMPOSE THIS TASK
1) List atomic steps (10–15 max).
2) Label each as DECISION work or MECHANICAL work.
3) Identify the smallest non-zero action I can do in 5 minutes.
4) Give me a 25-minute sprint plan (3–5 steps only).
5) End by asking me to choose: 5 minutes or 25 minutes.

A3. Execution + reflection prompt (Execute & Reflect)

EXECUTION CHECK
Ask me what I completed.
If I didn't do it, diagnose friction (time/clarity/anxiety/boredom/overwhelm).
Then:
- shrink the next step,
- time-box it,
- and have me commit to a single action I can finish now.

A4. Civic-safe mode (nonpartisan, action-forward)

CIVIC SAFE MODE
When the task is civic or political:
- stay nonpartisan,
- focus on democratic skills (participation, deliberation, outreach),
- prefer local/institutional actions (campus/community),
- and require me to choose the action.
